## 3. 优先排查路径（基于潜在风险的预防性排查）

尽管用例全部通过，鉴于 IoT 系统的复杂性，建议对以下“隐形风险点”进行人工或专项排查，以防自动化用例未能覆盖的边缘场景：

1.  **OTA 升级的异常中断恢复**：
    *   自动化通常模拟“成功升级”路径。需确认在升级过程中断电、断网、存储空间不足时，设备是否能正确回滚或进入安全模式。
2.  **权限管理的细粒度控制**：
    *   当前权限模块仅 3 条用例，可能仅覆盖了“管理员”和“普通用户”。需排查是否存在越权访问（IDOR）风险，例如普通用户通过修改 API 参数访问管理员接口。
3.  **日志数据的实时性与完整性**：
    *   验证日志上报是否存在延迟，以及在弱网环境下日志是否丢失。自动化断言通常只检查“是否有日志”，较少检查“日志内容是否完整且时序正确”。
4.  **设备状态同步的一致性**：
    *   在设备管理模块中，确认云端状态与设备本地状态在高频操作下是否最终一致（例如：快速开关设备 10 次，云端显示是否正确）。

## 4. 建议补充的回归用例

当前用例分布存在明显的**倾斜**，部分高风险模块用例过少。建议按以下优先级补充：

### 🔴 高优先级：OTA 固件升级模块 (当前 9 条)
OTA 是 IoT 设备最高风险的操作，一旦失败可能导致设备变砖。
*   **补充方向**：
    *   **版本回退测试**：升级失败后自动回退到上一版本的验证。
    *   **差分包升级**：验证增量升级（Diff Patch）的正确性，对比全量升级。
    *   **并发升级压力**：模拟大量设备同时请求升级包，验证服务器带宽和设备下载稳定性。
    *   **校验和失败**：故意篡改固件 MD5/SHA256，验证设备是否拒绝安装。

### 🟠 中优先级：权限管理模块 (当前 3 条)
3 条用例对于权限系统来说严重不足，极易遗漏安全漏洞。
*   **补充方向**：
    *   **RBAC 矩阵覆盖**：覆盖不同角色（Admin, Operator, Viewer）对同一资源的不同操作（CRUD）。
    *   **Token 过期与刷新**：验证 Access Token 过期后，Refresh Token 机制是否正常工作，以及非法 Token 的拦截。
    *   **水平/垂直越权**：尝试使用用户 A 的 Token 操作用户 B 的资源。

### 🟡 低优先级：日志管理模块 (当前 5 条)
*   **补充方向**：
    *   **日志级别过滤**：验证 DEBUG/INFO/WARN/ERROR 级别过滤是否生效。
    *   **敏感信息脱敏**：检查日志中是否包含密码、密钥、PII 等敏感信息（合规性要求）。

## 5. 对测试框架维护的建议

### 5.1 增强断言的深度 (Deep Assertion)
*   **现状**：目前可能多基于 HTTP 状态码（200 OK）或简单的 JSON 字段存在性进行断言。
*   **建议**：引入**Schema 校验**（如 `jsonschema`）确保返回数据结构符合契约；对于 OTA 和日志模块，增加**业务逻辑断言**（例如：升级后版本号必须严格大于旧版本，日志时间戳必须在操作时间之后）。

### 5.2 引入数据驱动测试 (Data-Driven Testing)
*   **现状**：OTA 和权限用例数量较少，可能存在硬编码。
*   **建议**：使用 `pytest.mark.parametrize` 结合 YAML/Excel 数据文件，将测试数据与代码分离。例如，用一份 CSV 文件定义 50 种不同的权限组合，自动生成 50 条用例，从而低成本扩充权限模块的覆盖率。

### 5.3 增加非功能性测试集成
*   **现状**：当前主要为功能测试。
*   **建议**：
    *   **性能监控**：在 Pytest 中集成 `pytest-benchmark` 或简单的时间断言，监控关键接口（如设备列表查询、OTA 包下载）的响应时间，防止性能回退。
    *   **稳定性测试**：为 OTA 模块增加重试机制和长时间运行的稳定性套件（Soak Test）。

### 5.4 Allure 报告优化
*   **建议**：
    *   为每个测试步骤添加详细的 `allure.step()`，特别是 OTA 升级过程中的“下载”、“校验”、“写入”、“重启”等子步骤，便于故障定位。
    *   附加设备日志片段或截图到 Allure 报告中，即使用例通过，也保留现场证据以便后续审计。
